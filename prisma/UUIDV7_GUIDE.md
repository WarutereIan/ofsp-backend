# UUIDv7 Implementation Guide

**Time-ordered UUIDs for better database performance**

---

## Overview

All IDs in the system now use **UUIDv7** (RFC 9562), which provides:

- ✅ **Time-ordered** - UUIDs are sortable by creation time
- ✅ **Better indexing** - Improves B-tree index performance
- ✅ **Database-friendly** - Reduces index fragmentation
- ✅ **Traceable** - Can extract timestamp from UUID

---

## Implementation

### PostgreSQL Function

The `gen_uuidv7()` function is created in **migration 001** and generates UUIDv7 as TEXT:

```sql
SELECT gen_uuidv7();
-- Returns: '018f1234-5678-7abc-def0-123456789abc'
```

### Usage in Tables

All table ID fields use `DEFAULT gen_uuidv7()`:

```sql
CREATE TABLE "users" (
  "id" TEXT NOT NULL DEFAULT gen_uuidv7(),
  ...
);
```

### Prisma Schema

The Prisma schema continues to use `@default(uuid())` for client generation:

```prisma
model User {
  id        String     @id @default(uuid())
  ...
}
```

**Note:** Prisma will generate UUIDs in the application layer, but the database defaults to UUIDv7 when IDs are not provided.

## Benefits

### 1. Better Index Performance

UUIDv7 is time-ordered, so new records are inserted at the end of indexes:

```sql
-- UUIDv4 (random) - causes index fragmentation
INSERT INTO users (id, ...) VALUES 
  ('a1b2c3d4-...', ...),  -- Random position
  ('z9y8x7w6-...', ...); -- Random position

-- UUIDv7 (time-ordered) - sequential inserts
INSERT INTO users (id, ...) VALUES 
  ('018f1234-...', ...),  -- End of index
  ('018f1235-...', ...);  -- End of index
```

### 2. Natural Sorting

UUIDs can be sorted by creation time:

```sql
SELECT * FROM users ORDER BY id;
-- Results are naturally sorted by creation time
```

### 3. Timestamp Extraction

You can extract the creation timestamp from UUIDv7:

```sql
-- Extract timestamp (milliseconds since Unix epoch)
SELECT 
  id,
  to_timestamp(
    ('x' || substring(id, 1, 8))::bit(32)::bigint * 1000 +
    ('x' || substring(id, 10, 4))::bit(16)::bigint
  )::timestamptz AS created_at
FROM users;
```

## Application Usage

### Prisma Client

Prisma will generate UUIDs when creating records:

```typescript
// Prisma generates UUID (v4 by default)
const user = await prisma.user.create({
  data: {
    email: 'user@example.com',
    // id is auto-generated by Prisma
  }
});
```

### Database Defaults

If you insert directly into the database without providing an ID:

```sql
-- ID will be generated using UUIDv7
INSERT INTO users (email, phone, ...) 
VALUES ('user@example.com', '+254...', ...);
```

## Performance Considerations

### Index Maintenance

- ✅ **UUIDv7:** New records append to index (minimal fragmentation)
- ⚠️ **UUIDv4:** Random inserts cause index fragmentation

### Storage

- Same storage size: 36 characters (TEXT) or 16 bytes (UUID type)
- No additional storage overhead

### Query Performance

- ✅ Better for range queries on creation time
- ✅ Better for pagination (cursor-based)
- ✅ Better for time-series analytics

## Future Enhancements

### PostgreSQL 18 Native Support

When upgrading to PostgreSQL 18, you can use the native function:

```sql
-- Replace custom function with native
DROP FUNCTION gen_uuidv7();
-- Use native uuidv7() instead
ALTER TABLE users ALTER COLUMN id SET DEFAULT uuidv7()::text;
```

### UUID Type

Consider using UUID type instead of TEXT:

```sql
-- Future migration
ALTER TABLE users ALTER COLUMN id TYPE UUID USING id::uuid;
ALTER TABLE users ALTER COLUMN id SET DEFAULT uuidv7();
```
